# tetris v1.2.2 — sustainable zoom & scaling

This is a small but architectural patch on top of [v1.2.1](https://github.com/paulfxyz/tetris/releases/tag/v1.2.1). The previous zoom mechanism — `transform: scale()` on the wrap + `clamp()` on the canvas's CSS size + an `aspect-ratio` override — produced subtle but real visual bugs: comb-like smear trails after zoom-out, and a stretch of empty space below the board at certain viewport heights because width and height clamps could disagree.

## 🩹 What changed

**One source of truth.** A new CSS variable `--board-h` (in real CSS pixels) is set on `.game-wrap`. Every dependent dimension — canvas width/height, HUD column widths, the grid gap — is derived from it in pure CSS:

```css
.game-wrap {
  --board-h: 600px;
  --board-w: calc(var(--board-h) / 2);
  --hud-w:   clamp(96px, calc(var(--board-h) * .22), 150px);
  --gap:     clamp(8px,  calc(var(--board-h) * .03),  22px);
}
#game-canvas { width: var(--board-w); height: var(--board-h); }
```

**No more `transform: scale()`.** The canvas changes size in real CSS pixels, so `getBoundingClientRect()` and the DPR-aware bitmap (`canvas.width × DPR`) are always coherent. The renderer's bitmap can't disagree with what the user sees.

**Device-aware fit.** `computeFittedHeight()` measures the actual stage rect, subtracts realistic reservations (mobile stacking, virtual pad), and clamps to `[360px, 920px]`. Manual `+` / `−` is a 60–160% multiplier on top of that base, so "Fit" then `+` composes correctly.

**Viewport resilience.** Listens to `resize` and `orientationchange`, so the board re-fits when the iOS Safari bar collapses, the on-screen keyboard appears, or the device rotates.

## What you'll notice

- No diagonal smear under the bottom row at low zoom.
- No empty white space below the lowest blocks at certain window heights.
- HUD blocks scale proportionally with the board instead of staying chunky on a tiny board.
- Fit-to-screen works on the first click. Zoom in/out preserves the user's choice.
